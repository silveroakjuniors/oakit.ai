# Requirements Document

## Introduction

The Franchise Model introduces a multi-school ownership layer above the existing single-school tenant model in Oakit. A Franchise Admin can own and manage multiple schools under a single franchise umbrella, upload shared curriculum that all member schools consume, view aggregate school-level statistics, and create new schools — all without ever accessing personally identifiable information (PII) belonging to students, teachers, or parents. The existing `school_id`-based tenant isolation remains the primary security boundary; the franchise layer sits above it as a read-aggregate and curriculum-sharing layer only.

---

## Glossary

- **Franchise**: A named entity that groups one or more schools under a single ownership umbrella.
- **Franchise_Admin**: A user with the `franchise_admin` role who owns and manages a franchise. Separate from `admin`, `principal`, and `super_admin` roles.
- **Member_School**: A school that belongs to a franchise, identified by its existing `school_id`.
- **Franchise_Curriculum**: A curriculum document uploaded by a Franchise_Admin that is shared across all Member_Schools within the same franchise.
- **Franchise_Chunk**: A curriculum chunk derived from a Franchise_Curriculum document, owned at the franchise level and readable by all Member_Schools.
- **Franchise_Class**: A class definition created by a Franchise_Admin that is shared as a template across all Member_Schools.
- **School_Plan**: A day plan generated by an individual Member_School's admin from Franchise_Chunks. Plans are scoped to the school and are not visible to the Franchise_Admin.
- **Aggregate_Stats**: School-level counts and rates (student count, teacher count, curriculum status, plan completion rate) that contain no individual PII.
- **PII**: Personally Identifiable Information — any data that can identify an individual, including names, phone numbers, email addresses, parent contacts, and home addresses of students, teachers, or parents.
- **Privacy_Notice**: A persistent UI banner displayed to school principals and admins informing them that their school's PII is not shared with the Franchise_Admin.
- **Consent_Agreement**: A formal data-sharing agreement that must be signed before any PII can be shared outside a school's tenant boundary.
- **Super_Admin**: The platform-level administrator who can see all schools and all franchises.
- **schoolScope**: The existing Express middleware that enforces `school_id` tenant isolation on all API routes.
- **PII_Guard**: A new middleware that intercepts requests from `franchise_admin` users to endpoints returning individual PII records and returns HTTP 403.
- **Teaching_Days_Service**: The service that computes net teaching days per school by subtracting holidays and special days from total working days.
- **Curriculum_Coverage_Fit**: An indicator (`exact`, `under`, `over`) comparing a school's net teaching days against the total franchise chunk count for a class.
- **Chunk_Summary_Format**: The human-readable table format (Week, Day, Topic, Preview) used to present franchise curriculum chunks to the Franchise_Admin for review and approval.
- **Net_Teaching_Days**: Total working days minus full-day holidays and special days, with half-days counting as 0.5.

## Requirements

### Requirement 1: Franchise Entity and School Association

**User Story:** As a Super_Admin, I want to create franchises and associate schools with them, so that the platform can model multi-school ownership structures.

#### Acceptance Criteria

1. THE System SHALL maintain a `franchises` table with fields: `id`, `name`, `contact_email`, `contact_phone`, `address`, `status`, and `created_at`.
2. THE System SHALL maintain a `franchise_memberships` table that records the association between a `franchise_id` and a `school_id`, with a unique constraint preventing a school from belonging to more than one franchise.
3. WHEN a Super_Admin creates a franchise, THE System SHALL require `name` and `contact_email` and return the created franchise record.
4. WHEN a Super_Admin associates a school with a franchise, THE System SHALL verify the school exists and is not already a member of another franchise before creating the membership.
5. IF a school is already a member of a franchise and a request is made to associate it with a different franchise, THEN THE System SHALL return HTTP 409 with the message "School is already a member of a franchise".
6. THE System SHALL preserve `school_id` as the primary tenant isolation key for all existing features; franchise membership SHALL NOT alter any existing `schoolScope` middleware behaviour.

---

### Requirement 2: Franchise Admin Role and Authentication

**User Story:** As a Super_Admin, I want to create Franchise_Admin accounts, so that franchise owners can log in and manage their franchise.

#### Acceptance Criteria

1. THE System SHALL support a `franchise_admin` role as a distinct value in the `role` column of the `users` table.
2. WHEN a `franchise_admin` user authenticates, THE JWT_Service SHALL issue a JWT containing `user_id`, `franchise_id`, `role: "franchise_admin"`, and `permissions`, with `school_id` set to `null`.
3. THE schoolScope middleware SHALL permit `franchise_admin` requests to pass through without a `school_id` check, in the same manner as `super_admin`.
4. WHEN a `franchise_admin` JWT is presented to any route that is not explicitly authorised for `franchise_admin`, THE System SHALL return HTTP 403 with the message "Insufficient permissions".
5. THE System SHALL allow a Super_Admin to create a `franchise_admin` user and bind that user to exactly one franchise via a `franchise_id` foreign key on the `users` table.
6. IF a `franchise_admin` user attempts to access a route scoped to a `school_id` that is not a member of their franchise, THEN THE System SHALL return HTTP 403 with the message "Access denied: school is not a member of your franchise".

---

### Requirement 3: Franchise Curriculum Sharing

**User Story:** As a Franchise_Admin, I want to upload curriculum PDFs that are automatically available to all my member schools, so that curriculum is centrally owned and consistently distributed.

#### Acceptance Criteria

1. WHEN a Franchise_Admin uploads a curriculum PDF, THE Ingestion_Service SHALL create a `curriculum_documents` record with `franchise_id` set and `school_id` set to `null`, indicating franchise-level ownership.
2. WHEN the Ingestion_Service processes a franchise curriculum document, THE Ingestion_Service SHALL create `curriculum_chunks` records with `franchise_id` set and `school_id` set to `null`.
3. WHEN a Member_School's admin or principal queries curriculum chunks for a class, THE Curriculum_Service SHALL return both school-owned chunks (where `school_id` matches) and franchise-owned chunks (where `franchise_id` matches the school's franchise), merged and ordered by `chunk_index`.
4. THE Franchise_Admin SHALL be able to upload curriculum scoped to a Franchise_Class, not to a school-specific class.
5. WHEN a Franchise_Admin edits a Franchise_Chunk's `topic_label` or `content`, THE System SHALL update the chunk record and the change SHALL be immediately visible to all Member_Schools.
6. IF a Member_School's admin attempts to delete or overwrite a Franchise_Chunk, THEN THE System SHALL return HTTP 403 with the message "Franchise-owned chunks cannot be modified by school admins".
6a. IF a Member_School's admin attempts to delete a Franchise_Curriculum document, THE System SHALL return HTTP 403 with the message "Franchise-owned curriculum cannot be deleted by school admins".
6b. IF a Member_School's admin attempts to re-ingest or replace a curriculum document that is franchise-owned, THE System SHALL return HTTP 403 with the message "Franchise-owned curriculum cannot be replaced by school admins".
6c. THE Chunk_Guard middleware SHALL enforce the delete/modify restriction at the API layer on ALL chunk and curriculum document mutation endpoints (`DELETE`, `PUT`, `PATCH`) — the restriction SHALL apply regardless of whether the request originates from the UI or a direct API call.
6d. WHEN a Member_School's admin views the curriculum management page, THE Frontend SHALL hide the delete and re-upload controls for franchise-owned chunks and documents, and display a read-only badge labelled "Managed by Franchise".
6e. THE System SHALL enforce the franchise chunk protection rule for ALL users scoped to a Member_School — including `admin`, `principal`, and `teacher` roles — not only for admin users.
6f. THE Chunk_Guard middleware SHALL only apply the franchise protection rules to schools that have a `franchise_memberships` record — schools onboarded directly (independent schools with no franchise association) SHALL retain full control over their own curriculum documents and chunks, including the ability to delete, re-upload, and re-ingest without restriction.
7. THE System SHALL allow each Member_School to generate its own School_Plans from Franchise_Chunks independently; plan records SHALL be scoped to `school_id` and SHALL NOT be visible to the Franchise_Admin.

### Requirement 4: School Creation by Franchise Admin

**User Story:** As a Franchise_Admin, I want to create new schools under my franchise, so that I can expand my franchise without requiring Super_Admin intervention for each school.

#### Acceptance Criteria

1. WHEN a Franchise_Admin submits a school creation request with `name`, `plan_type`, and `contact`, THE System SHALL create a new school record and automatically create a `franchise_memberships` record linking the new school to the Franchise_Admin's franchise.
2. THE System SHALL enforce no upper limit on the number of schools a Franchise_Admin may create.
3. WHEN a Franchise_Admin creates a school, THE System SHALL generate a `subdomain` from the school name using the same slug logic applied by the Super_Admin school creation endpoint.
4. IF a school with the same `name` already exists, THEN THE System SHALL return HTTP 409 with the message "School with this name already exists".
5. WHEN a school is created by a Franchise_Admin, THE System SHALL record `created_by_franchise_admin: true` and the `franchise_id` on the school record so Super_Admins can identify franchise-created schools.

---

### Requirement 5: Franchise Dashboard — Aggregate Stats

**User Story:** As a Franchise_Admin, I want to view school-level aggregate statistics for all my member schools, so that I can monitor performance without accessing individual PII.

#### Acceptance Criteria

1. WHEN a Franchise_Admin requests the franchise dashboard, THE Dashboard_Service SHALL return one summary record per Member_School containing: `school_id`, `school_name`, `school_address`, `school_contact` (phone and email of the school entity), `total_students`, `total_teachers`, `curriculum_status`, and `plan_completion_rate`.
2. THE Dashboard_Service SHALL compute `total_students` as a COUNT of student records for the school and SHALL NOT include any student name, phone, email, or other identifying field.
3. THE Dashboard_Service SHALL compute `total_teachers` as a COUNT of teacher records for the school and SHALL NOT include any teacher name, phone, email, or other identifying field.
4. THE Dashboard_Service SHALL compute `curriculum_status` as the ingestion status of the most recently uploaded curriculum document for each Member_School.
5. THE Dashboard_Service SHALL compute `plan_completion_rate` as the percentage of scheduled day plans marked complete for the current academic period, expressed as a number between 0 and 100.
6. WHEN a Franchise_Admin requests stats for a `school_id` that is not a member of their franchise, THE System SHALL return HTTP 403 with the message "Access denied: school is not a member of your franchise".

---

### Requirement 6: PII Protection Enforcement at the API Layer

**User Story:** As a school principal, I want assurance that my school's student, teacher, and parent data is never accessible to the Franchise_Admin, so that my school's data remains proprietary.

#### Acceptance Criteria

1. THE PII_Guard middleware SHALL inspect the `role` claim in the JWT of every inbound request; WHEN the role is `franchise_admin`, THE PII_Guard SHALL evaluate whether the requested endpoint returns individual PII records.
2. WHEN a `franchise_admin` request targets any endpoint that returns individual student records, THE System SHALL return HTTP 403 with the body `{ "error": "PII data is proprietary to each school" }`.
3. WHEN a `franchise_admin` request targets any endpoint that returns individual teacher records, THE System SHALL return HTTP 403 with the body `{ "error": "PII data is proprietary to each school" }`.
4. WHEN a `franchise_admin` request targets any endpoint that returns individual parent records, THE System SHALL return HTTP 403 with the body `{ "error": "PII data is proprietary to each school" }`.
5. THE PII_Guard SHALL enforce the restriction at the middleware layer independently of any UI-level access controls, so that the restriction cannot be bypassed by direct API calls.
6. THE System SHALL permit `franchise_admin` requests to endpoints that return only aggregate counts, school entity fields (`name`, `address`, `contact_phone`, `contact_email`), or franchise-owned curriculum data.
7. IF a new API route is added that returns student, teacher, or parent records, THE System SHALL require explicit opt-in to allow `franchise_admin` access; the default SHALL be denied.
8. THE PII_Guard SHALL log every blocked `franchise_admin` PII request to the audit log with fields: `timestamp`, `franchise_admin_user_id`, `franchise_id`, `attempted_endpoint`, `http_method`.

---

### Requirement 7: School-Side Privacy Notice

**User Story:** As a school principal, I want to see a clear notice in my dashboard confirming that my school's PII is not shared with the franchise owner, so that I can trust the platform and communicate this to my staff and parents.

#### Acceptance Criteria

1. WHEN a user with role `admin` or `principal` whose school is a Member_School loads any authenticated page, THE Frontend SHALL display a persistent Privacy_Notice banner.
2. THE Privacy_Notice banner SHALL contain the exact text: "No personal information (students, teachers, parents) is shared with your franchise owner. This data is proprietary to your school. If you need to enable data sharing, please contact the system admin. A consent agreement must be signed before any data is shared outside your school."
3. THE System SHALL expose a GET endpoint `/api/v1/schools/:school_id/franchise-privacy-status` that returns `{ is_franchise_member: boolean, privacy_notice_text: string }` so the frontend can conditionally render the banner.
4. WHEN a school is not a member of any franchise, THE System SHALL return `{ is_franchise_member: false }` and THE Frontend SHALL NOT display the Privacy_Notice banner.
5. THE Privacy_Notice banner SHALL be non-dismissible; it SHALL remain visible on every page load for franchise member schools.

---

### Requirement 8: Super Admin Visibility of Franchise Structure

**User Story:** As a Super_Admin, I want to see which schools belong to which franchise, so that I can audit the franchise structure and manage the platform holistically.

#### Acceptance Criteria

1. WHEN a Super_Admin requests the schools list, THE System SHALL include a `franchise_id` and `franchise_name` field on each school record where applicable, and `null` for independent schools.
2. THE System SHALL expose a GET endpoint `/api/v1/super-admin/franchises` that returns all franchises with fields: `id`, `name`, `contact_email`, `contact_phone`, `address`, `status`, `school_count`, `created_at`.
3. THE System SHALL expose a GET endpoint `/api/v1/super-admin/franchises/:franchise_id/schools` that returns the list of Member_Schools for a given franchise, including `school_id`, `school_name`, `school_status`, `plan_type`, and `joined_at`.
4. WHEN a Super_Admin requests `/api/v1/super-admin/franchises/:franchise_id/schools`, THE System SHALL return school entity fields only and SHALL NOT include student, teacher, or parent records.
5. THE System SHALL allow a Super_Admin to remove a school from a franchise by deleting the `franchise_memberships` record, without deleting the school itself.
6. THE System SHALL allow a Super_Admin to deactivate a franchise, which SHALL set the franchise `status` to `inactive` but SHALL NOT deactivate the Member_Schools.

---

### Requirement 9: Consent Model for Future PII Sharing

**User Story:** As a Super_Admin, I want a consent framework in place so that if a school ever agrees to share PII with its franchise owner, the agreement is formally recorded and auditable.

#### Acceptance Criteria

1. THE System SHALL maintain a `franchise_consent_agreements` table with fields: `id`, `franchise_id`, `school_id`, `agreed_by_user_id`, `agreed_at`, `consent_scope` (JSON array of permitted data categories), `status` (`pending` | `active` | `revoked`), and `document_url`.
2. WHEN no active consent agreement exists for a school-franchise pair, THE PII_Guard SHALL enforce the default-deny rule for `franchise_admin` PII requests regardless of any other configuration.
3. WHEN an active consent agreement exists for a school-franchise pair and the requested data category is listed in `consent_scope`, THE System SHALL permit the `franchise_admin` request to proceed, subject to Super_Admin enabling the override.
4. THE System SHALL require Super_Admin approval before any consent agreement transitions from `pending` to `active`.
5. WHEN a consent agreement is revoked, THE System SHALL immediately re-apply the default-deny PII rule for that school-franchise pair without requiring a server restart.
6. THE System SHALL record every consent agreement status change in the audit log with fields: `timestamp`, `changed_by_user_id`, `franchise_id`, `school_id`, `old_status`, `new_status`.

### Requirement 10: Correctness Properties for PII Isolation

**User Story:** As a platform engineer, I want verifiable correctness properties for the PII isolation rules, so that regressions in access control are caught automatically.

#### Acceptance Criteria

1. FOR ALL API endpoints in the system, WHEN the endpoint is called with a valid `franchise_admin` JWT and the endpoint returns individual student, teacher, or parent records, THE System SHALL return HTTP 403 — this property SHALL hold for every such endpoint without exception.
2. FOR ALL `franchise_admin` requests that are blocked by PII_Guard, THE System SHALL return the body `{ "error": "PII data is proprietary to each school" }` — the error body SHALL be identical across all blocked endpoints.
3. FOR ALL aggregate stat responses returned to a `franchise_admin`, THE System SHALL return zero fields whose name matches the pattern `*_name`, `*_phone`, `*_email`, `*_address` at the individual record level — only school-entity-level contact fields are permitted.
4. WHEN a `franchise_admin` JWT is issued, THE JWT SHALL contain `school_id: null`; FOR ALL such tokens, THE schoolScope middleware SHALL NOT reject the request on the basis of a missing `school_id`.
5. FOR ALL schools that are members of franchise F, WHEN the franchise dashboard endpoint is called with a valid `franchise_admin` JWT for franchise F, THE System SHALL return exactly one summary record per member school — no more, no fewer.
6. FOR ALL schools that are NOT members of franchise F, WHEN the franchise dashboard endpoint is called with a valid `franchise_admin` JWT for franchise F, THE System SHALL return zero records for those schools.
7. WHEN a Franchise_Chunk is updated by a Franchise_Admin, FOR ALL Member_Schools of that franchise, THE Curriculum_Service SHALL return the updated chunk content in subsequent chunk queries — the update SHALL be reflected consistently across all member schools.
8. FOR ALL consent agreement records with `status = 'pending'` or `status = 'revoked'`, THE PII_Guard SHALL deny `franchise_admin` PII requests for the corresponding school-franchise pair — the default-deny rule SHALL be enforced regardless of the presence of a consent record.

### Requirement 11: Franchise Class Creation and Curriculum Upload

**User Story:** As a Franchise_Admin, I want to create classes and upload curriculum to each class in a clear, understandable format, so that all member schools receive consistent, well-structured teaching content.

#### Acceptance Criteria

1. WHEN a Franchise_Admin creates a class, THE System SHALL store the class as a `Franchise_Class` record with fields: `id`, `franchise_id`, `name` (e.g. "UKG", "LKG", "Grade 1"), `description`, and `created_at` — with no `school_id`, making it franchise-owned.
2. WHEN a Franchise_Admin uploads a curriculum PDF to a Franchise_Class, THE Ingestion_Service SHALL process the document and create `Franchise_Chunk` records scoped to `franchise_id` and `franchise_class_id`.
3. WHEN chunks are created from a franchise curriculum, THE System SHALL present each chunk to the Franchise_Admin in a human-readable summary format containing: `chunk_index`, `topic_label`, `week_number` (derived from chunk position), `day_number` (derived from chunk position within the week), and a `content_preview` (first 200 characters of the chunk content).
4. THE Franchise_Admin dashboard SHALL display the full chunk list for each Franchise_Class as a structured table with columns: Week, Day, Topic, Preview — so the franchise owner can verify the curriculum is correctly parsed before it is distributed to schools.
5. WHEN a Franchise_Admin reviews the chunk list and approves it, THE System SHALL mark the curriculum document `status = 'approved'` and make the chunks available to all Member_Schools.
6. WHEN a Franchise_Admin rejects or re-uploads a curriculum, THE System SHALL replace the existing chunks for that Franchise_Class with the newly processed chunks and reset the `status` to `pending_review`.
7. WHEN a Member_School queries chunks for a class that matches a Franchise_Class name, THE Curriculum_Service SHALL return the approved Franchise_Chunks for that class alongside any school-specific chunks, merged and ordered by `chunk_index`.
8. THE System SHALL allow a Franchise_Admin to edit the `topic_label` of any Franchise_Chunk after approval; the edit SHALL propagate immediately to all Member_Schools without requiring re-approval.
9. THE System SHALL display the total chunk count per Franchise_Class to the Franchise_Admin as: `total_chunks`, `total_weeks` (ceil(total_chunks / 5)), and `estimated_teaching_days` (equal to `total_chunks` assuming one chunk per teaching day).

### Requirement 12: Teaching Days Calculation per School

**User Story:** As a Franchise_Admin, I want to see the total number of actual teaching days for each member school after they configure their academic calendar, so that I can understand how much of the shared curriculum each school can realistically cover.

#### Acceptance Criteria

1. WHEN a Member_School's admin configures their academic calendar (working days, start date, end date, holidays, settling days, sports days, red days, and other special days), THE Teaching_Days_Service SHALL compute the `net_teaching_days` for that school as:
   `net_teaching_days = total_working_days − full_day_special_days − holidays`
   where half-day special days count as 0.5 teaching days.
2. THE Franchise_Admin dashboard SHALL display a `teaching_days` column in the school summary table showing each Member_School's `net_teaching_days` for the current academic year.
3. WHEN a Member_School has not yet configured their academic calendar, THE System SHALL display `"Calendar not set"` for that school's teaching days field in the franchise dashboard — no count SHALL be shown.
4. THE Teaching_Days_Service SHALL break down the deductions per school as: `total_working_days`, `holidays_count`, `settling_days_count`, `sports_days_count`, `red_days_count`, `other_special_days_count`, `half_days_count`, and `net_teaching_days` — all as aggregate counts with no individual event names or dates exposed to the Franchise_Admin.
5. WHEN a Franchise_Admin views a school's teaching days detail, THE System SHALL show only the counts breakdown (e.g. "3 settling days", "2 sports days", "5 holidays") — it SHALL NOT show the names or dates of individual events, as those may contain school-specific information.
6. THE System SHALL also display a `curriculum_coverage_fit` indicator per school: `"exact"` if `net_teaching_days = total_franchise_chunks`, `"under"` if `net_teaching_days > total_franchise_chunks` (more days than content), or `"over"` if `net_teaching_days < total_franchise_chunks` (more content than days).
7. WHEN a Member_School updates their calendar (adds or removes holidays or special days), THE Teaching_Days_Service SHALL recompute `net_teaching_days` and the updated value SHALL be reflected in the franchise dashboard within the next dashboard refresh.
8. THE System SHALL expose a GET endpoint `/api/v1/franchise/schools/:school_id/teaching-days` accessible only to `franchise_admin` users whose franchise owns the school, returning the counts breakdown and `net_teaching_days` — individual event names and dates SHALL NOT be included in the response.

### Requirement 13: Curriculum Coverage Visibility for Franchise Admin

**User Story:** As a Franchise_Admin, I want to see how well each school's teaching days align with the shared curriculum chunks, so that I can identify schools that need calendar adjustments or additional curriculum content.

#### Acceptance Criteria

1. THE Franchise_Admin dashboard SHALL display a per-school curriculum coverage summary with: `school_name`, `net_teaching_days`, `total_franchise_chunks`, `curriculum_coverage_fit`, and `chunks_per_day` (total_franchise_chunks ÷ net_teaching_days, rounded to 2 decimal places).
2. WHEN `curriculum_coverage_fit` is `"over"` (more chunks than teaching days), THE System SHALL display a visual indicator (e.g. amber warning) so the Franchise_Admin can identify schools at risk of not completing the curriculum.
3. WHEN `curriculum_coverage_fit` is `"under"` (more teaching days than chunks), THE System SHALL display a visual indicator (e.g. blue info) so the Franchise_Admin knows the curriculum will cycle or have gaps.
4. WHEN `curriculum_coverage_fit` is `"exact"`, THE System SHALL display a green indicator.
5. THE System SHALL NOT expose individual day plan details, teacher names, or any other school-internal data in the coverage summary — only the aggregate counts and fit indicator are permitted.
