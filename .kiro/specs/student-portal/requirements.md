# Requirements Document

## Introduction

The Student Portal is a new module for the Oakit platform that gives students (kids) direct access to their learning data. It covers six areas: admin configuration to enable the portal per class, student account management, a student-facing portal for daily topics and homework, an AI doubts assistant scoped strictly to covered topics, a quiz/test system with teacher-assigned and self-initiated tests, and analytics visible to students, parents, and teachers.

The portal integrates with existing Oakit data: `daily_completions`, `curriculum_chunks`, `teacher_homework`, `homework_submissions`, `students`, `sections`, `classes`, `schools`, and the existing JWT auth system. The AI doubts and quiz features extend the existing Python AI service (`query_pipeline.py`).

## Glossary

- **Student_Portal**: The web interface accessible to students after login.
- **Student_Account**: A login credential set (username + password) generated for a student.
- **Student**: A child enrolled in a class whose class has the Student Portal enabled.
- **Admin**: The school administrator who manages school setup and portal access.
- **Class_Teacher**: The teacher assigned as the primary teacher for a section.
- **Supporting_Teacher**: A teacher assigned to a section in a non-primary capacity.
- **Parent**: A guardian linked to a student via `parent_student_links`.
- **Section**: A class division (e.g., LKG-A) linked to students and teachers.
- **Portal_Config**: The per-class configuration record that enables or disables the Student Portal.
- **Covered_Topics**: The set of `curriculum_chunks` marked complete in `daily_completions` for a section up to and including the current date.
- **Daily_Feed**: The student-facing view showing topics covered, notes, and homework for a selected date.
- **Homework_History**: The student-facing view showing homework entries and submission statuses across a date range.
- **AI_Doubts**: The AI assistant available to students, scoped strictly to Covered_Topics.
- **Quiz**: A set of questions generated by AI or a teacher, covering a specified subject and date range.
- **Assigned_Test**: A Quiz created and assigned to students by a teacher, with a time limit and no AI assistance during the attempt.
- **Self_Test**: A Quiz initiated by a student for self-practice, with AI no assistance available.
- **Quiz_Result**: The stored record of a student's answers, marks, and AI evaluation for a completed Quiz.
- **Analytics_Dashboard**: The view showing per-student test history, marks, progress over time, and weak area identification.
- **Credential_Generator**: The system component that creates and resets Student_Account credentials.
- **Username**: A unique identifier for a student in the format `firstname-classname-section` (e.g., `aarav-lkg-a`), lowercased and hyphenated.
- **Default_Password**: The initial password `123456` assigned to every new Student_Account.

## Requirements

### Requirement 1: Admin Configuration — Enable Student Portal Per Class

**User Story:** As an admin, I want to enable or disable the Student Portal for specific classes, so that only students in enabled classes can log in and use the portal.

#### Acceptance Criteria

1. THE Admin SHALL be able to enable or disable the Student Portal for each class (e.g., LKG, UKG) independently via the admin settings panel.
2. WHEN the admin enables the Student Portal for a class, THE Portal_Config SHALL record the enabled state and the timestamp for that class within the school.
3. WHEN the admin disables the Student Portal for a class, THE Portal_Config SHALL record the disabled state and students of that class SHALL be unable to log in to the Student_Portal.
4. THE Admin SHALL be able to view the current enabled/disabled status of the Student Portal for all classes on a single settings page.
5. IF a student attempts to log in and their class has the Student Portal disabled, THEN THE Student_Portal SHALL reject the login and display the message: "Your school has not enabled the student portal for your class."
6. WHEN the admin changes the portal status for a class, THE Portal_Config SHALL take effect immediately without requiring a server restart.

---

### Requirement 2: Student Account Management

**User Story:** As an admin or class teacher, I want to generate and manage login credentials for students, so that each student has a unique, secure account to access the portal.

#### Acceptance Criteria

1. THE Credential_Generator SHALL create a Student_Account for a student with a Username in the format `{firstname}-{classname}-{section}`, all lowercase and hyphenated (e.g., `aarav-lkg-a`).
2. WHEN a Username collision is detected, THE Credential_Generator SHALL append a numeric suffix to produce a unique Username (e.g., `aarav-lkg-a-2`).
3. THE Credential_Generator SHALL set the Default_Password `123456` for every newly created Student_Account.
4. WHEN an admin or Class_Teacher generates credentials for a student, THE Credential_Generator SHALL store a bcrypt-hashed version of the password and SHALL NOT store the plaintext password.
5. THE Admin and Class_Teacher SHALL be able to reset a student's password to the Default_Password `123456` at any time.
6. WHEN a student logs in for the first time using the Default_Password, THE Student_Portal SHALL prompt the student to change their password before accessing any other feature.
7. WHEN a student changes their password, THE Student_Portal SHALL require the new password to be at least 6 characters long.
8. THE Admin and Class_Teacher SHALL be able to generate credentials for all students in a section in a single bulk operation.
9. IF a Student_Account already exists for a student and credentials are regenerated, THEN THE Credential_Generator SHALL reset the password to Default_Password and SHALL NOT create a duplicate account.
10. THE Admin SHALL be able to generate credentials only for students whose class has the Student Portal enabled.

---

### Requirement 3: Student Portal — Daily Feed and Navigation

**User Story:** As a student, I want to see what topics were covered on any school day and navigate between dates, so that I can review my learning and stay up to date.

#### Acceptance Criteria

1. WHEN a student logs in, THE Student_Portal SHALL display the Daily_Feed for today's date by default.
2. THE Daily_Feed SHALL show all Covered_Topics for the selected date, grouped by subject, with the topic name and any teacher notes.
3. THE Student_Portal SHALL allow the student to navigate to any previous school date to view the Daily_Feed for that date.
4. THE Student_Portal SHALL allow the student to navigate up to 5 school days ahead of today's date.
5. WHEN a student selects a date that has no Covered_Topics, THE Daily_Feed SHALL display the message: "Nothing was covered on this date."
6. THE Daily_Feed SHALL display the homework entry for the selected date if one exists for the student's section.
7. THE Student_Portal SHALL display a Homework_History view showing homework entries and the student's submission status for the last 30 school days.
8. WHEN a student views the Homework_History, THE Homework_History SHALL show each homework entry with its date, content, and submission status (completed, partial, or not submitted).
9. THE Student_Portal SHALL provide a summary view showing all Covered_Topics for a user-selected date range.
10. THE Student_Portal SHALL provide a "Till Today" summary showing all Covered_Topics from the start of the current academic term up to and including today.
11. WHILE a student is viewing a date in the future beyond 5 school days, THE Student_Portal SHALL display the message: "Topics for this date are not yet available."

### Requirement 4: AI Doubts — Scoped to Covered Topics

**User Story:** As a student, I want to ask questions about topics I have already studied, so that I can get clear explanations without being distracted by topics I haven't covered yet.

#### Acceptance Criteria

1. THE AI_Doubts assistant SHALL only answer questions that are relevant to the student's Covered_Topics.
2. WHEN a student submits a question, THE AI_Doubts assistant SHALL check whether the question is relevant to the student's Covered_Topics before generating a response.
3. IF a student asks a question that is not relevant to any Covered_Topic, THEN THE AI_Doubts assistant SHALL respond: "I can only help with topics your class has covered. Try asking about [list up to 3 recent covered topics]."
4. WHEN a student asks a relevant question, THE AI_Doubts assistant SHALL provide a clear, age-appropriate explanation referencing the specific covered topic.
5. THE AI_Doubts assistant SHALL include the topic name and the date it was covered in every response, so the student can locate it in the Daily_Feed.
6. THE AI_Doubts assistant SHALL NOT provide answers to questions about topics not present in the student's Covered_Topics, regardless of how the question is phrased.
7. IF the AI service is unavailable, THEN THE AI_Doubts assistant SHALL display: "Oakie is unavailable right now. Please try again shortly." and SHALL NOT display a blank screen.
8. THE AI_Doubts assistant SHALL be disabled during an Assigned_Test attempt.

### Requirement 5: Quiz and Test System

**User Story:** As a student or teacher, I want to generate and take quizzes based on covered topics, so that students can practice and teachers can assess learning.

#### Acceptance Criteria

1. WHEN a student requests a Self_Test for a subject, THE Student_Portal SHALL display the Covered_Topics for that subject within the student's chosen date range and ask the student to confirm or edit the topic list before generating questions.
2. WHEN a teacher creates an Assigned_Test, THE Student_Portal SHALL display the Covered_Topics for the selected subject and date range to the teacher and allow the teacher to add or remove topics before generating questions.
3. BEFORE generating any Quiz, THE Student_Portal SHALL show the final confirmed topic list to the requester (student or teacher) and require explicit confirmation.
4. WHEN generating Quiz questions, THE AI_Doubts assistant SHALL generate questions only from the confirmed topic list and SHALL NOT include topics outside that list.
5. THE Quiz SHALL support the following question types: fill-in-the-blank, descriptive, 1-mark, and 2-mark.
6. WHEN a student initiates a Self_Test, THE Student_Portal SHALL ask the student to select the desired question types before generating the Quiz.
7. WHEN a teacher creates an Assigned_Test, THE Student_Portal SHALL allow the teacher to select the question types and set a time limit of 15 minutes, 30 minutes, or 60 minutes.
8. WHILE a student is taking an Assigned_Test, THE AI_Doubts assistant SHALL be disabled and the student SHALL NOT be able to access the Daily_Feed or any other portal section.
9. WHEN the time limit of an Assigned_Test expires, THE Student_Portal SHALL automatically submit the student's current answers.
10. WHEN a student submits a Quiz, THE AI_Doubts assistant SHALL evaluate the answers, assign marks, and store the Quiz_Result including the student's answers, correct answers, marks awarded, and total marks.
11. AFTER a Quiz is submitted, THE Student_Portal SHALL display the Quiz_Result to the student, including correct answers and an explanation for each incorrect answer.
12. THE Student_Portal SHALL identify subjects and chapters where the student scored below 50% and display a prompt: "You may want to revise [subject/chapter]. Tap to review covered topics."
13. WHEN a teacher assigns a test to a section, THE Student_Portal SHALL notify all students in that section that a new Assigned_Test is available, including the subject, time limit, and due date.
14. THE Quiz questions SHALL be generated with sufficient randomness so that two students requesting a quiz on the same topic and date range receive different question sets.
15. IF a student attempts to access AI_Doubts or navigate away during an Assigned_Test, THEN THE Student_Portal SHALL display a warning and block the navigation.

### Requirement 6: Analytics

**User Story:** As a student, parent, or teacher, I want to see test performance analytics, so that learning progress and weak areas can be identified and addressed.

#### Acceptance Criteria

1. THE Analytics_Dashboard SHALL display, for each student, the total number of quizzes taken, average marks scored, and a progress chart showing marks over time.
2. THE Analytics_Dashboard SHALL identify and display the subjects and chapters where the student's average score is below 50%, labeled as "Needs Revision."
3. WHEN a teacher views the Analytics_Dashboard for their section, THE Analytics_Dashboard SHALL display individual student analytics for all students in that section.
4. THE Analytics_Dashboard SHALL be accessible to the Class_Teacher and Supporting_Teacher of the student's section only; teachers of other sections SHALL NOT be able to view it.
5. WHEN a parent views the Analytics_Dashboard for their child, THE Analytics_Dashboard SHALL display the child's quiz history, marks, and weak area summary.
6. IF a student's class does not have the Student Portal enabled, THEN THE Analytics_Dashboard SHALL NOT be visible to that student's parent.
7. THE Analytics_Dashboard SHALL update within 60 seconds of a Quiz_Result being stored.
8. THE Student SHALL be able to view their own Analytics_Dashboard after completing at least one Quiz.
9. WHEN a teacher views the Analytics_Dashboard, THE Analytics_Dashboard SHALL allow the teacher to filter by subject, date range, and individual student.
10. THE Analytics_Dashboard SHALL display a class-level summary showing the average score per subject across all students in the section.

---

### Requirement 7: Access Control

**User Story:** As a system, I want to enforce role-based access so that students, parents, and teachers can only access data they are authorized to see.

#### Acceptance Criteria

1. THE Student_Portal SHALL authenticate students using JWT tokens issued upon successful login with a valid Student_Account.
2. WHEN a student's JWT token expires, THE Student_Portal SHALL redirect the student to the login page and SHALL NOT display any protected content.
3. THE Student_Portal SHALL only display data belonging to the student's own section and school; a student SHALL NOT be able to access another student's data.
4. THE Class_Teacher and Supporting_Teacher SHALL only be able to generate credentials, assign tests, and view analytics for students in their own assigned sections.
5. THE Admin SHALL be able to generate credentials and view analytics for any student in the school.
6. WHEN a parent accesses the Analytics_Dashboard, THE Analytics_Dashboard SHALL display data only for the parent's linked children via `parent_student_links`.
7. IF a student's class has the Student Portal disabled after the student has already logged in, THEN THE Student_Portal SHALL invalidate the student's active session and redirect to the login page with the message: "Access to the student portal has been disabled for your class."
8. THE Student_Portal SHALL enforce rate limiting on login attempts: after 5 failed attempts within 10 minutes, THE Student_Portal SHALL block further login attempts from that IP for 15 minutes.
9. THE Student_Portal SHALL use a separate JWT token type (`role: "student"`) distinct from the existing teacher/admin/parent tokens to prevent privilege escalation.
